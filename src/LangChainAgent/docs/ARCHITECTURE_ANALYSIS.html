১. বিদ্যমান সিস্টেম বিশ্লেষণ

১.১ Proxi Bridge এর বর্তমান Agent সিস্টেম

Proxi Bridge এ ইতিমধ্যে তিনটি agent-related service আছে:

সার্ভিসফাইলকাজমডেল

১.২ DiskRAGService (ragService.ts) - বিস্তারিত

এটি একটি disk-based RAG সিস্টেম যা নিচের কাজগুলো করে:

  • setWorkingDirectory(dir) - কাজের ফোল্ডার সেট করে, SSOT.md তৈরি করে
  • scanProject() - প্রজেক্ট scan করে, ফাইল, ডিপেন্ডেন্সি খুঁজে বের করে
  • generateSSOTTemplate() - SSOT.md তৈরি করে (Project Structure, Dependencies, Source Files, Config, Docs)
  • readSSOT() / saveSSOT() / appendToSSOT() - SSOT পড়া/লেখা/আপডেট
  • searchSSOT(query) - keyword-based section search
  • addToSession(msg) / getSessionContext() - session buffer (২০টি message পর্যন্ত)
  • grantPermission() / hasPermission() - scope-based permission system

৪. Response Normalizer - বিভিন্ন মডেলের Format একত্রীকরণ

৪.১ সমস্যা

গুরুত্বপূর্ণ: বিভিন্ন OpenAI-compatible মডেল ভিন্ন ভিন্ন format এ response দেয়:

Groq: { choices: [{ message: { content: "..." } }] }
OpenCode Zen: { choices: [{ message: { content: "..." } }] } (কিন্তু কখনো streaming chunks আলাদা)
Ollama: { message: { content: "..." } } (choices নেই!)
Gemini: { candidates: [{ content: { parts: [{ text: "..." }] } }] }
Anthropic: { content: [{ text: "..." }] }

সুতরাং, মডেল switch করলে response parse fail হতে পারে!

৪.২ ResponseNormalizer Class

export class ResponseNormalizer { // যেকোনো format থেকে standard content বের করা static extractContent(raw: any, provider?: string): string { if (!raw) return ''; // 1. Groq / OpenAI standard format if (raw.choices?.[0]?.message?.content) { return raw.choices[0].message.content; } // 2. Ollama format (no choices array) if (raw.message?.content && !raw.choices) { return raw.message.content; } // 3. Gemini format if (raw.candidates?.[0]?.content?.parts?.[0]?.text) { return raw.candidates[0].content.parts[0].text; } // 4. Anthropic format if (Array.isArray(raw.content) && raw.content[0]?.text) { return raw.content[0].text; } // 5. Direct string if (typeof raw === 'string') return raw; // 6. Fallback: try JSON.stringify return JSON.stringify(raw); } // Usage/token normalize করা static extractUsage(raw: any): { prompt: number; completion: number; total: number } { if (raw.usage) { return { prompt: raw.usage.prompt_tokens || 0, completion: raw.usage.completion_tokens || 0, total: raw.usage.total_tokens || 0, }; } // Ollama format if (raw.prompt_eval_count || raw.eval_count) { return { prompt: raw.prompt_eval_count || 0, completion: raw.eval_count || 0, total: (raw.prompt_eval_count || 0) + (raw.eval_count || 0), }; } return { prompt: 0, completion: 0, total: 0 }; } // Stream chunks normalize করা static normalizeStreamChunk(chunk: any): string { if (chunk.choices?.[0]?.delta?.content) { return chunk.choices[0].delta.content; } if (chunk.message?.content) { return chunk.message.content; } if (chunk.candidates?.[0]?.content?.parts?.[0]?.text) { return chunk.candidates[0].content.parts[0].text; } return ''; } }

৯.৪ Dependency Safety

কোনো version conflict নেই

  • নতুন কোড শুধু src/services/ এ যোগ হয়েছে
  • কোনো নতুন npm package install হয়নি
  • বিদ্যমান fs, path, crypto ব্যবহার করা হয়েছে
  • LangChainAgent মডিউল (src/LangChainAgent/) আলাদা থাকে
  • proxi_new সার্ভিস (src/services/) আলাদা থাকে
  • শুধু import path দিয়ে সংযুক্ত — zero dependency overlap

৮. Self-Healing Agent Pattern - SSOT Auto-Bootstrap

মূল ধারণা: প্রতিটি agent request এ, agent প্রথমে চেক করবে SSOT.md আছে কিনা। না থাকলে নিজেই তৈরি করবে। এতে করে agent কখনো ভুল করবে না।

৮.১ সমস্যা (বর্তমান অবস্থা)

বর্তমানে:
- ফাইল চৌকিদার (WorkspaceWatcher) ঘুমিয়ে পড়লে SSOT পুরনো থাকে
- নতুন ফাইল যোগ হলে agent জানে না
- SSOT না থাকলে agent ভুল উত্তর দেয়
- ক্লায়েন্ট (JetBrains/MCP) থেকে request এলেও SSOT check হয় না

৮.৩ SSOT State Machine

interface SSOTState { directory: string; flag: 0 | 1; // 0 = তৈরি হয়নি, 1 = তৈরি হয়েছে lastScan: number; // শেষ scan এর timestamp lastModified: number; // SSOT শেষ আপডেট fileCount: number; // ফাইল সংখ্যা (পরিবর্তন ট্র্যাক) contentHash: string; // SSOT content hash (integrity check) } // State Transitions: // State 0 (Uninitialized) → [createSSOT] → State 1 (Initialized) // State 1 (Initialized) → [fileChanged] → State 2 (Stale) // State 2 (Stale) → [rescan] → State 1 (Initialized)

৮.৪ ensureSSOT() Function

async function ensureSSOT( dir: string, rag: DiskRAGService ): Promise<string> { // 1. State check const state = getState(dir); if (state.flag === 1) { // SSOT আছে - কিন্তু কি আপডেট দরকার? const currentFileCount = await countFiles(dir); if (currentFileCount !== state.fileCount) { // নতুন ফাইল যোগ/মুছে ফেলা হয়েছে console.log(`📂 Files changed in ${dir}. Rescanning...`); await rescanAndUpdateSSOT(dir, rag); updateState(dir, { fileCount: currentFileCount }); } return rag.searchSSOT(''); // পুরো SSOT পড়ো } // 2. SSOT নেই - তৈরি করো console.log(`🆕 No SSOT found for ${dir}. Creating...`); const { needsPermission } = await rag.setWorkingDirectory(dir, { autoInit: true }); if (needsPermission) { throw new Error('Permission required for directory access'); } const fileCount = await countFiles(dir); setState(dir, { flag: 1, lastScan: Date.now(), fileCount, }); console.log(`✅ SSOT created for ${dir} with ${fileCount} files`); return rag.readSSOT(); }

৮.৭ সুবিধাগুলো

আগে (Watchdog dependent)

  • Watchdog থামলে SSOT পুরনো
  • Agent ভুল context ব্যবহার করে
  • Manual rescan দরকার
  • Client request এ SSOT check হয় না
  • State tracking নেই

পরে (Self-Healing)

  • Watchdog independent
  • Agent নিজেই SSOT maintain করে
  • Auto-rescan on file change
  • প্রতিটি request এ check
  • Persistent state tracking
  • Zero config, zero maintenance

৮.৮ Performance Consideration

প্রশ্ন: প্রতিটি request এ SSOT check করলে slow হবে না?

উত্তর: না! কারণ:
  • State check: In-memory Map lookup (0.001ms)
  • File count check: fs.readdirSync (1-5ms)
  • SSOT read: fs.readFileSync (1-2ms)
  • Total overhead: 2-7ms (negligible)
  • AI response time: 500-2000ms
সুতরাং, overhead মাত্র 0.3-1%!
